## When Good Traffic Looks Like an Outage

The first time our lending platform felt "down" during a healthy business day, the dashboards told a confusing story. CPU was fine. Database connections were stable. Error rates were low. Yet mobile clients timed out, support tickets spiked, and our queue depth climbed anyway.

The culprit was not a deploy—it was **abuse-shaped load**: credential stuffing, scripted quote refreshes, and a partner that retried without backoff. Cheap requests in bulk starved the pools real users needed.

That changed how I think about reliability. Availability is not only surviving hardware failure—it is **protecting capacity from work that should never be admitted**. Rate limits and request signing became first-class SRE controls, not a late security bolt-on.

## Why Rate Limits Belong in the SRE Toolkit

On-call engineers live with shared budgets: pools, worker concurrency, downstream quotas, and attention during incidents. Rate limiting enforces those budgets at the front door.

At a fintech startup I co-founded—400k+ users on Node.js and Redis—I treated limits as **capacity contracts**:

- **Protect the user path** before expensive middleware runs.
- **Make abuse visible** with distinct metrics (`429` by route, client, and key).
- **Fail predictably** instead of degrading into ambiguous 504s.

> A rate limit is a reliability boundary. Security is often a side effect.

IP blocks helped with obvious botnets; the durable win was **per-identity throttling** on OTP, KYC, and repayment routes.

## Layering Limits: Edge, API, and Business Rules

One global "1000 req/min" rule rarely works. I use three layers:

1. **Edge / gateway** — coarse IP and ASN throttles, WAF rules, geographic anomalies.
2. **API middleware** — authenticated user or API-key buckets on hot routes.
3. **Domain guards** — business rules ("max 3 OTP sends per hour per phone") enforced in the service layer.

Redis-backed sliding windows worked well for us because counters were fast, TTLs were natural, and we could tune limits per route without redeploying app servers.

```typescript
import { Redis } from "ioredis";
import { randomUUID } from "crypto";

const redis = new Redis(process.env.REDIS_URL!);

export async function slidingWindowAllow(
  key: string,
  limit: number,
  windowSec: number
): Promise<{ allowed: boolean; remaining: number }> {
  const now = Date.now();
  const windowStart = now - windowSec * 1000;

  // Member must be unique; two hits in the same ms would otherwise collapse into one.
  const member = `${now}-${randomUUID()}`;

  const pipe = redis.multi();
  pipe.zremrangebyscore(key, 0, windowStart);
  pipe.zadd(key, now, member);
  pipe.zcard(key);
  pipe.expire(key, windowSec);
  const [, , count] = (await pipe.exec())!.map((r) => r[1]) as number[];

  const allowed = count <= limit;
  return { allowed, remaining: Math.max(0, limit - count) };
}
```

Middleware stayed thin: resolve a limit key, call the window helper, return `429` with `Retry-After` when blocked.

```typescript
export function rateLimit(limit: number, windowSec: number, keyFn: (req: Request) => string) {
  return async (req: Request, res: Response, next: NextFunction) => {
    const key = `rl:${keyFn(req)}`;
    const { allowed, remaining } = await slidingWindowAllow(key, limit, windowSec);
    res.setHeader("X-RateLimit-Remaining", String(remaining));
    if (!allowed) {
      res.setHeader("Retry-After", String(windowSec));
      return res.status(429).json({ error: "rate_limit_exceeded" });
    }
    next();
  };
}
```

The important operational detail: **different routes get different keys**. Collapsing everything into one bucket punishes legitimate power users while still letting attackers pivot routes.

## Request Signing as a Contract, Not Secret Sauce

Rate limits slow anonymous floods. **Request signing** stops forged or replayed calls on machine-to-machine paths—webhooks, mobile backends talking to internal APIs, partner settlement callbacks.

I do not treat signing as "encrypt the body." I treat it as an **integrity and freshness contract**:

- Canonical string over method, path, timestamp, and hashed body.
- HMAC with a rotated shared secret (or asymmetric keys for larger partner fleets).
- Clock skew window (e.g., ±5 minutes) to block replays.

```typescript
import crypto from "crypto";

export function signRequest(
  secret: string,
  method: string,
  path: string,
  body: string,
  timestamp: string
): string {
  const bodyHash = crypto.createHash("sha256").update(body).digest("hex");
  const payload = `${method}\n${path}\n${timestamp}\n${bodyHash}`;
  return crypto.createHmac("sha256", secret).update(payload).digest("hex");
}

export function verifySignedRequest(
  secret: string,
  req: { method: string; path: string; body: string; headers: Record<string, string | undefined> }
): boolean {
  const timestamp = req.headers["x-timestamp"];
  const signature = req.headers["x-signature"];
  if (!timestamp || !signature) return false;

  const skewMs = Math.abs(Date.now() - Date.parse(timestamp));
  if (skewMs > 5 * 60 * 1000) return false;

  const expected = signRequest(secret, req.method, req.path, req.body, timestamp);
  const sigBuf = Buffer.from(signature);
  const expBuf = Buffer.from(expected);

  // timingSafeEqual throws on length mismatch; reject forged-length sigs cleanly.
  if (sigBuf.length !== expBuf.length) return false;
  return crypto.timingSafeEqual(sigBuf, expBuf);
}
```

On partner callbacks, malformed payloads stopped masquerading as app bugs, and spikes on signed routes pointed to real partners or rotated keys—not random spray.

## Wiring Controls into Observability and Incident Response

Controls only matter if you can **see them work**. We added New Relic and CloudWatch dashboards that split:

- `429` rate by route and limit tier
- Signature verification failures (bad clock, bad body hash, unknown key id)
- P95 latency on throttled vs unthrottled routes

During a later flood, limits absorbed the blast while autoscaling caught up; signing blocked webhook replays that would have double-posted ledger rows. MTTR improved because on-call asked "Which bucket is full?" instead of "Is the app broken?"

## Key Takeaways

- **Reliability and abuse are the same problem at scale**—unbounded work consumes the capacity your paying users need.
- **Layer rate limits** at edge, API, and domain levels; one global cap is rarely enough.
- **Use Redis sliding windows** for fast, tunable per-route budgets with clear `429` semantics.
- **Treat request signing as freshness + integrity**, with canonical payloads, timestamps, and `timingSafeEqual` verification.
- **Instrument limits and signature failures** as SRE signals, not security-only logs.

Rate limits and signing will not replace WAFs, bot management, or thoughtful architecture. They will, however, turn chaotic overload into **bounded, observable failure**—which is exactly what you want when reliability meets abuse.
