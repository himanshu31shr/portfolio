## The Wake-Up Call

At a fintech startup I co-founded, we hit 400K users faster than our infrastructure matured. One Wednesday morning our API gateway started returning 503s across the board. The culprit wasn't a traffic spike from real users — it was a single actor hammering our loan eligibility endpoint at 12,000 requests per minute with scripted credential-stuffing payloads. Our Node.js services buckled, Redis connections pooled up, and legitimate users couldn't even check their balances.

That incident changed how I think about reliability. **Rate limiting and request signing aren't just security features — they're SRE controls.** They sit at the intersection of availability, integrity, and abuse prevention. Without them, your uptime SLO is one bad actor away from a breach.

## Why Traditional Rate Limiting Falls Short

Most teams start with a naive approach: a global rate limiter in the API gateway that caps requests per IP. We did too. The problem is that IP-based limits break under shared networks (entire office buildings behind one NAT), mobile carriers rotating IPs, and attackers using rotating proxy pools.

What we needed was **layered, context-aware rate limiting** — different limits for different operations, applied at multiple levels: per-user, per-endpoint, and per-action sensitivity.

Here's the sliding window implementation we settled on using Redis, deployed behind our Node.js services:

```typescript
import Redis from 'ioredis';

const redis = new Redis(process.env.REDIS_URL);

interface RateLimitResult {
  allowed: boolean;
  remaining: number;
  retryAfter?: number;
}

async function slidingWindowRateLimit(
  key: string,
  maxRequests: number,
  windowMs: number
): Promise<RateLimitResult> {
  const now = Date.now();
  const windowStart = now - windowMs;

  const pipeline = redis.pipeline();
  pipeline.zremrangebyscore(key, 0, windowStart);
  pipeline.zadd(key, now.toString(), `${now}-${Math.random()}`);
  pipeline.zcard(key);
  pipeline.pexpire(key, windowMs);

  const results = await pipeline.exec();
  const currentCount = results![2][1] as number;

  return {
    allowed: currentCount <= maxRequests,
    remaining: Math.max(0, maxRequests - currentCount),
    retryAfter: currentCount > maxRequests ? windowMs / 1000 : undefined,
  };
}
```

The sliding window approach using Redis sorted sets gave us precise control without the burst problems of fixed-window counters. We applied different thresholds per endpoint tier: **100 req/min** for read-heavy endpoints like balance checks, **10 req/min** for sensitive actions like loan applications, and **3 req/min** for OTP generation to prevent SMS bombing.

## Request Signing: Authenticating Intent, Not Just Identity

Rate limiting tells you *how much* a client is asking. Request signing tells you *whether the client is who they claim to be* — and whether the request was tampered with in transit. For a financial platform where every API call could move money, this distinction was critical.

We implemented HMAC-based request signing where every client received a secret key, and every request had to include a signature computed over the request body, timestamp, and URI:

```typescript
import crypto from 'crypto';

interface SignedRequestHeaders {
  'x-signature': string;
  'x-timestamp': string;
  'x-client-id': string;
}

function generateSignature(
  clientSecret: string,
  method: string,
  path: string,
  timestamp: string,
  body: string
): string {
  const payload = `${method}\n${path}\n${timestamp}\n${body}`;
  return crypto
    .createHmac('sha256', clientSecret)
    .update(payload)
    .digest('hex');
}

function verifyRequest(
  headers: SignedRequestHeaders,
  method: string,
  path: string,
  body: string,
  getSecret: (clientId: string) => string
): boolean {
  const { 'x-signature': signature, 'x-timestamp': timestamp, 'x-client-id': clientId } = headers;

  // Reject requests older than 5 minutes to prevent replay attacks
  const requestAge = Date.now() - parseInt(timestamp, 10);
  if (requestAge > 5 * 60 * 1000) return false;

  const expectedSignature = generateSignature(
    getSecret(clientId),
    method,
    path,
    timestamp,
    body
  );

  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(expectedSignature)
  );
}
```

Two details matter here. First, the **timestamp check** prevents replay attacks — an attacker can't capture a valid signed request and reuse it hours later. Second, `crypto.timingSafeEqual` prevents timing-based side-channel attacks where an attacker could deduce correct signature bytes by measuring response times.

## Tying It Together: The Middleware Stack

In production, these controls composed into a middleware pipeline that ran before any business logic. The order was deliberate: **IP blocklist → request signing verification → rate limiting → handler**. We checked the blocklist first because it's the cheapest operation — a single Redis `SISMEMBER` call. Signature verification came next to reject forged requests before they consumed rate limit quota. Only authenticated, non-blocked requests counted against rate limits.

```typescript
// Express middleware composition
app.use('/api/v1',
  ipBlocklistMiddleware,     // O(1) Redis lookup
  requestSigningMiddleware,  // HMAC verification
  rateLimitMiddleware({      // Sliding window check
    tiers: {
      default: { max: 100, windowMs: 60_000 },
      sensitive: { max: 10, windowMs: 60_000 },
      otp: { max: 3, windowMs: 60_000 },
    }
  }),
  router
);
```

This ordering prevented a subtle bug we hit early on: without signature verification before rate limiting, attackers could exhaust a legitimate user's rate limit by forging requests with their client ID.

## Measuring the Impact

After deploying these controls, the numbers told the story. We integrated the middleware metrics into our monitoring stack and tracked three key indicators:

- **Blocked abusive requests**: 23% of all inbound traffic was identified and rejected as automated abuse — traffic that previously consumed compute and degraded latency for real users.
- **Critical production anomalies**: Dropped by **65%** over the following quarter, measured through our APM tooling. The reduction came primarily from eliminating cascading failures triggered by abuse-driven resource exhaustion.
- **Platform uptime**: Stabilized at **99.9%** as measured over a rolling 90-day window, up from roughly 99.2% before the controls were in place.

The most surprising insight was that rate limiting improved our **p99 latency** by 40ms. By shedding abusive traffic early in the middleware stack, we freed connection pool slots and event loop capacity for legitimate requests.

## Key Takeaways

1. **Rate limiting is an SRE primitive, not just a security feature.** If your uptime SLO doesn't account for abuse traffic, you're measuring availability under ideal conditions that production will never give you.

2. **Layer your limits by sensitivity.** A single global rate limit is a blunt instrument. Tier your endpoints — read operations can tolerate higher throughput, but sensitive mutations need tight caps.

3. **Request signing prevents quota theft.** Without verifying *who* is making a request before counting it, attackers can weaponize your own rate limiter against legitimate users.

4. **Middleware order matters.** Cheapest checks first, most expensive last. Blocklist → signing → rate limit → handler. Getting this wrong creates subtle vulnerabilities and wastes compute.

5. **Measure the reliability impact, not just the security impact.** Track how abuse mitigation affects your p99 latency, error rates, and connection pool utilization — the SRE metrics that matter for user experience.

These controls aren't glamorous. They don't show up in feature demos or sprint reviews. But when your platform is serving hundreds of thousands of users in a regulated financial environment, the difference between 99.2% and 99.9% uptime is the difference between a platform users trust and one they abandon.
